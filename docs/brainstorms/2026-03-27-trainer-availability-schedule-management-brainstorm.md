# Trainer Availability Schedule Management Brainstorm

## What We're Building
트레이너가 직접 자신의 예약 가능 시간을 관리할 수 있는 기능을 도입한다. 이번 단계의 목적은 `예약 가능한 실제 슬롯 생성/편집`을 트레이너에게 바로 개방하는 것이 아니라, `트레이너 개인 가능 시간표 + 휴무/예외 일정`을 self-service 방식으로 관리하게 만드는 것이다.

초기 버전은 예약 도메인에 과도하게 개입하지 않고, 운영자가 참고할 수 있는 신뢰 가능한 가용성 데이터 레이어를 만드는 데 집중한다. 이 정보는 당장은 조회/운영 참고용으로 쓰되, 이후 예약 슬롯 생성 정책의 기준으로 확장 가능해야 한다.

## Why This Needs Deliberate Scoping
현재 저장소의 `TrainerSchedule`은 예약 생성/정원 검증에 쓰이는 슬롯 데이터다. 이것을 그대로 트레이너 self-service CRUD로 열면, 권한·정원·중복·취소 책임이 한 번에 열리면서 범위가 급격히 커진다.

반대로 실제 현장 요구는 먼저 `언제 근무 가능하고 언제 쉬는지`를 트레이너가 스스로 반영할 수 있게 해 달라는 쪽에 가깝다. 따라서 초기 단계는 `개인 가능 시간표 관리`를 먼저 만들고, 이 데이터를 나중에 예약 슬롯 생성 기준으로 연결하는 순서가 더 안전하다.

## Approaches Considered
### Approach A: 트레이너가 실제 예약 슬롯을 직접 생성/편집
- PT/GX 슬롯, 시작/종료 시각, 정원 등을 트레이너가 직접 등록/수정한다.
- 장점: 예약 화면과 바로 연결되고 기능 체감이 빠르다.
- 단점: 현재 예약 도메인 정합성 규칙과 운영 충돌 위험이 크다.
- 판단: 초기 버전으로는 과하다.

### Approach B: 날짜별 일정만 직접 입력
- 트레이너가 매일 근무 가능/휴무를 날짜 단위로 모두 입력한다.
- 장점: 개념이 단순하다.
- 단점: 반복되는 주간 패턴을 계속 수동 입력해야 해 운영 비용이 크다.
- 판단: 실제 사용성이 떨어진다.

### Approach C (Selected): 기본 주간 시간표 + 날짜별 예외 관리
- 기본 주간 가능 시간을 반복 규칙으로 저장한다.
- 특정 날짜에는 `휴무` 또는 `단일 시간 범위 override + 메모`를 저장한다.
- UI는 캘린더를 포함하되, 데이터 모델은 `반복 규칙 + 날짜 예외` 구조로 유지한다.
- 장점: 운영 현실과 데이터 구조 균형이 가장 좋다.
- 단점: 캘린더 UI와 예외 규칙 정의가 필요하다.

## Selected Direction
초기 버전은 아래 방향으로 고정한다.

1. 트레이너는 `내 스케줄` 화면에서 본인 개인 가능 시간표를 관리한다.
2. 기본 데이터 모델은 `주간 반복 가능 시간표`다.
3. 특정 날짜에는 아래 중 하나만 설정 가능하다.
   - 기본 시간표 따름
   - 하루 전체 휴무
   - 단일 시간 범위 override + 메모
4. 한 날짜에 여러 시간 구간은 초기 범위에서 제외한다.
5. 이 정보는 초기에는 조회/운영 참고용이지만, 나중에 예약 슬롯 생성 기준으로 확장 가능해야 한다.

## Roles and Visibility
초기 권한 모델은 아래로 제한한다.

- `ROLE_TRAINER`: 본인 시간표/휴무 조회 및 편집 가능
- `ROLE_CENTER_ADMIN`, `ROLE_DESK`: 트레이너 시간표/휴무 조회 가능
- `ROLE_CENTER_ADMIN`, `ROLE_DESK`: 초기 버전에서는 편집 불가

이렇게 하면 트레이너 self-service 목적은 살리되, 운영 주체가 일정 상태를 참고할 수 있다. 초기부터 관리자 override 편집까지 열면 변경 책임과 조정 충돌이 생기기 쉬우므로 이번 범위에서는 제외한다.

## UX Direction
초기 UI는 `트레이너 전용 내 스케줄 화면`으로 두는 것이 맞다. 예약 관리 화면이나 트레이너 관리 화면에 섞으면 기능 책임이 흐려진다.

추천 화면 구조:
- 상단: 기본 주간 가능 시간 설정
- 본문: 월간 캘린더
- 날짜 클릭 시: 예외 일정 편집 패널/모달
- 보조 영역: 해당 월 예외 일정 목록

중요한 점은 사용자는 캘린더 중심으로 체감하되, 내부 모델은 `주간 기본 시간표 + 날짜별 예외`라는 것이다.

## Exception Policy
초기 예외 정책은 단순하게 유지한다.

- 한 날짜에는 하나의 예외 설정만 허용
- 예외 종류는 `휴무` 또는 `단일 시간 범위 override`
- 예외에는 메모를 남길 수 있음
- 다중 시간 구간, 오전/오후 분할 근무, 같은 날짜 복수 예외는 이번 범위에서 제외

이 제한이 중요한 이유는 일정 겹침 검증, 정렬, 부분 공백 시간 처리, 이후 예약 슬롯 연결 규칙을 크게 단순화하기 때문이다.

## Data Semantics
이 데이터는 초기에는 `참고용`으로 사용하되, 의미 자체는 `향후 예약 슬롯 생성의 기준이 될 수 있는 데이터`로 정의해야 한다.

즉:
- 지금 당장은 관리자/데스크가 “이 트레이너가 언제 가능한지” 확인하는 운영 정보
- 이후에는 “이 가능 시간 밖으로는 예약 슬롯을 만들지 않는다”는 정책으로 확장 가능한 구조

이 방향을 택하면 지금은 범위를 통제하면서도, 나중에 다시 모델을 뒤엎지 않아도 된다.

## Risks To Avoid
- 현재 `TrainerSchedule` 슬롯 도메인을 그대로 self-service 편집 대상으로 오해하는 것
- 캘린더 UI를 넣는다는 이유로 초기부터 다중 시간 구간까지 여는 것
- 관리자/데스크 편집 권한까지 같이 열어 변경 책임이 흐려지는 것
- 참고용 데이터로 시작하면서도 향후 슬롯 생성 기준으로 확장할 여지를 남기지 않는 것
- 예약 슬롯 자동 생성/승인까지 이번 범위에 섞어 범위를 무너뜨리는 것

## Recommended Delivery Slice
### Slice 1
- 트레이너 전용 `내 스케줄` 진입점 정의
- 권한/조회 범위 정책 고정
- 기본 주간 가능 시간표 모델 정의

### Slice 2
- 월간 캘린더 + 특정 날짜 예외 등록 UX 정의
- `휴무`, `단일 시간 범위 override`, `메모` 흐름 고정
- 관리자/데스크 조회 surface 정의

### Slice 3
- 이 데이터와 예약 슬롯 생성 정책의 연결 지점을 문서로 정리
- 후속 단계에서 자동 슬롯 생성 또는 관리자 생성 기준으로 확장 여부 결정

## Success Criteria
- 트레이너가 본인 가능 시간과 휴무를 직접 관리할 수 있다.
- 관리자/데스크가 트레이너별 가능 시간과 예외 일정을 조회할 수 있다.
- 기본 모델은 `주간 반복 시간표 + 날짜별 예외`로 고정된다.
- 한 날짜에는 `기본 따름 / 하루 휴무 / 단일 시간 override + 메모` 중 하나만 적용된다.
- 기능이 아직 예약 슬롯 self-service 생성으로 번지지 않는다.
- 이후 예약 슬롯 생성 기준으로 확장 가능한 구조가 유지된다.

## Open Questions
- 초기 버전에서 관리자/데스크가 트레이너 시간표를 어떤 화면에서 조회하는 것이 가장 자연스러운가?
- 예외 메모는 내부 운영 메모로만 볼지, 향후 회원 노출 가능성까지 고려할지 정책을 분리할 필요가 있는가?
- 후속 단계에서 예약 슬롯 생성 주체를 트레이너로 둘지, 관리자/시스템으로 둘지 별도 결정이 필요한가?

## Resolved Questions
- 실제 예약 슬롯 직접 관리보다 `근무 가능 시간/휴무 관리`가 초기 범위에 더 적합하다.
- 표현 방식은 `기본 주간 시간표 + 날짜별 예외` 혼합형이 적절하다.
- 예외는 `휴무 + 특정 날짜 근무시간 변경 + 메모`까지 허용한다.
- 초기에는 한 날짜당 하나의 설정만 가능하도록 제한한다.
- 권한은 `트레이너 편집`, `관리자/데스크 조회 전용`이 적절하다.
- UI는 트레이너 전용 `내 스케줄` 화면으로 분리하는 것이 낫다.
- 이 데이터는 초기에는 참고용으로 쓰되, 구조는 향후 예약 슬롯 생성 기준으로 확장 가능해야 한다.
