# PT Availability-Based Reservation Brainstorm

## What We're Building
GX 예약은 기존처럼 이미 생성된 확정 스케줄 슬롯에 회원을 할당하는 구조를 유지한다. 반면 PT 예약은 트레이너의 개인 availability를 기준으로, 운영자가 실제 예약 시점에 1회성 예약을 생성하는 흐름을 추가한다.

초기 버전의 목표는 PT를 완전 자유 입력 예약으로 푸는 것이 아니라, `트레이너 가용시간 안에서만 생성 가능한 60분 블록 예약`으로 제한하는 것이다. 이렇게 하면 이미 만들어 둔 availability 기능과 기존 reservation lifecycle을 연결하면서도 운영 규칙을 단순하게 유지할 수 있다.

## Why This Approach
현재 reservation 도메인은 `trainer_schedules` 같은 확정 슬롯을 기준으로 동작한다. GX는 이 모델과 잘 맞지만, PT는 “미리 정해진 클래스”가 아니라 “트레이너와 회원이 조율해 잡는 1:1 일정”에 가깝다. 그래서 GX와 PT를 같은 예약 진입 방식으로 처리하면 PT 운영 현실이 잘 반영되지 않는다.

반대로 PT를 너무 자유롭게 열어 버리면 예약 시간 충돌, 차감 대상 불일치, 권한 범위 혼선이 커진다. 따라서 초기 버전은 `availability는 예약 후보 계산 기준`, `확정 예약은 기존 reservation 도메인에 기록`이라는 이중 구조를 유지하는 편이 가장 안전하다.

## Approaches Considered
### Approach A: PT도 기존처럼 미리 생성된 슬롯에만 예약
- 장점: 기존 reservation 구조를 그대로 재사용할 수 있다.
- 단점: PT 운영 현실과 맞지 않고, 데스크나 트레이너가 실제 조율 후 바로 잡기 어렵다.
- 판단: 현장 유연성이 부족하다.

### Approach B: PT를 완전 자유 시간 입력으로 예약
- 장점: 가장 유연하다.
- 단점: 시작 시각, 길이, 중복, 차감 정책이 한 번에 복잡해진다.
- 판단: 초기 범위로는 과하다.

### Approach C (Selected): availability 기반 예약 후보 계산 후 확정 예약 생성
- 트레이너와 날짜를 고르면 시스템이 가능한 시작 시각 목록을 계산해 보여준다.
- 사용자가 그중 하나를 선택하면 60분 PT 예약을 확정 생성한다.
- 장점: 운영 현실과 정합성 규칙 사이 균형이 가장 좋다.
- 단점: availability와 existing reservation 사이 연결 규칙 정의가 필요하다.

## Selected Direction
초기 PT 예약은 아래 규칙으로 고정한다.

1. GX는 기존 확정 슬롯 예약 구조를 유지한다.
2. PT는 트레이너 availability를 기준으로 예약 후보 시간을 계산한다.
3. PT 예약은 항상 `60분 블록`이다.
4. 실제 수업은 통상 50분으로 보더라도, 남은 10분은 정리/기록/전환 버퍼로 간주한다.
5. 시작 시각은 `30분 단위`로 허용한다.
6. 화면은 `트레이너 선택 -> 날짜 선택 -> 가능한 시작 시각 목록 선택` 흐름으로 제공한다.
7. 예약이 확정되면 기존 reservation 도메인 안의 확정 예약으로 기록돼야 한다.

## Reservation Eligibility
PT 예약 가능 대상은 강하게 제한한다.

- 활성 PT 회원권이 있어야 한다.
- 해당 회원권의 남은 횟수가 있어야 한다.
- 담당 트레이너 기준이 있는 경우, trainer actor는 본인 담당 회원권만 다룰 수 있어야 한다.

이 정책은 “예약은 잡혔는데 차감할 회원권이 없다”는 상태를 원천적으로 줄이기 위해 필요하다.

## Roles and Permissions
권한은 비대칭으로 둔다.

- `ROLE_CENTER_ADMIN`, `ROLE_DESK`
  - 모든 PT 예약 생성/변경/취소 가능
- `ROLE_TRAINER`
  - 본인 담당 회원의 PT 예약만 생성/변경/취소 가능
  - 본인 availability 범위 밖 예약 생성 불가
- GX 예약 생성 정책은 이번 변경 범위에 포함하지 않는다.

이렇게 하면 운영 주체는 데스크/관리자 쪽에 두면서도, 실제 회원과 조율하는 트레이너도 필요한 범위 안에서 업무를 처리할 수 있다.

## Time and Conflict Policy
PT 예약 시간 규칙은 단순하게 유지한다.

- 길이: 항상 60분
- 시작 시각: 30분 단위
- 트레이너 availability 범위 안에서만 생성 가능
- 과거 시각 예약 생성/변경 불가
- 트레이너 시간 중복 예약 금지
- 회원의 다른 예약과 시간 중복 금지

즉, 트레이너 availability가 `10:00-13:00`이면 가능한 시작 시각은 `10:00`, `10:30`, `11:00`, `11:30`, `12:00`처럼 60분 블록이 완전히 들어가는 시각만 후보가 된다.

## UX Direction
초기 UI는 자유형 캘린더 편집보다 계산형 선택 UX가 더 적합하다.

추천 흐름:
- 회원 선택
- PT 회원권 선택 또는 자동 선택
- 트레이너 선택
- 날짜 선택
- 시스템이 가능한 시작 시각 목록 계산
- 시작 시각 선택 후 예약 생성

이 방식은 availability를 사용자가 직접 해석하게 하지 않고, 시스템이 실제 예약 가능한 후보만 보여주므로 실수 가능성을 줄인다.

## Data Semantics
availability는 “확정 예약”이 아니라 “예약 가능 후보 계산의 기준 데이터”다. PT 예약 생성 시에는 이 availability를 읽어 가능한 시작 시각을 계산하고, 최종적으로는 확정 예약 레코드를 만들어 기존 예약 수명주기에 편입시켜야 한다.

즉:
- availability = 근무 가능 시간/휴무 데이터
- PT reservation = 실제 운영 예약 기록

이 경계를 유지해야 GX 슬롯 모델과 PT 동적 생성 모델을 함께 가져가도 도메인이 무너지지 않는다.

## Risks To Avoid
- availability를 예약 슬롯 자체로 취급해 예약/정원/취소 책임을 섞는 것
- PT 길이를 회원권/예약마다 다르게 열어 복잡도를 급격히 높이는 것
- 트레이너에게 본인 담당 범위를 넘는 PT 예약 권한을 주는 것
- 활성 PT 회원권/잔여 횟수 검증 없이 예약을 먼저 허용하는 것
- 회원/트레이너 중복 시간 검증을 UI에만 두고 서버에서 보장하지 않는 것

## Recommended Delivery Slice
### Slice 1
- PT 예약 정책 고정
- availability 기반 후보 시각 계산 규칙 정의
- 예약 eligibility 및 권한 경계 고정

### Slice 2
- PT 예약 생성 UX를 `트레이너 + 날짜 -> 가능한 시각 목록` 구조로 설계
- 관리자/데스크와 트레이너 actor별 대상 범위 차등 적용

### Slice 3
- 확정 예약 생성과 기존 reservation lifecycle 연결
- 시간 충돌, 회원권 잔여 횟수, actor scope 검증 테스트 추가

## Success Criteria
- GX는 기존 예약 흐름을 유지한다.
- PT는 availability 기반으로 예약 생성할 수 있다.
- PT 예약은 60분 고정, 30분 단위 시작 규칙을 따른다.
- 활성 PT 회원권과 남은 횟수가 있는 회원만 예약 가능하다.
- 데스크/관리자는 전체 PT 예약 가능, 트레이너는 본인 담당 회원만 가능하다.
- 회원과 트레이너의 시간 겹침이 모두 차단된다.
