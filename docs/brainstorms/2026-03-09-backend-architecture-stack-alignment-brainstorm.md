# Backend Architecture Stack Alignment

## What We're Building

`docs/02_시스템_아키텍처_설계서.md`의 Backend 기술스택 정의에 맞춰 현재 백엔드를 재정렬하는 작업을 검토했다. 이번 브레인스토밍의 대상은 단순 문서 수정이 아니라, 실제 코드베이스를 아키텍처 문서가 전제하는 방향으로 옮기는 것이다.

1차 목표는 `JPA/QueryDSL + OpenAPI` 도입이다. 현재의 `JdbcClient` 중심 저장소 구조를 일반 도메인 전반으로 점진 전환하고, 인증 영역도 같은 전환 범위에 포함한다. 반면 Redis는 이번 1차 범위에서 제외한다. 즉, 이번 단계는 저장소 기술과 API 문서화 계층을 먼저 정렬하고, 런타임 캐시/락/토큰 저장소 전환은 다음 단계로 미룬다.

## Why This Approach

검토한 선택지는 세 가지였다.

- 문서만 현실화하고 코드 전환은 하지 않는 방안
- `JPA/QueryDSL + OpenAPI`만 우선 도입하고 Redis는 제외하는 방안
- 문서에 맞춰 `JPA/QueryDSL + Redis + OpenAPI`를 한 번에 모두 도입하는 방안

선택한 방향은 두 번째다.

이유는 명확하다. 현재 코드베이스에서 가장 큰 문서-구현 차이는 저장소 접근 방식과 OpenAPI 부재다. 반면 Redis는 이미 다른 실행계획 문서에서 "현 단계 보류"로 정리된 적이 있고, 인증/QR/예약의 상태 저장 방식을 함께 바꾸면 리스크가 급격히 커진다. 따라서 이번 단계는 JPA/QueryDSL과 OpenAPI를 중심으로 실제 구현을 이동시키고, Redis는 별도 전환 단계로 분리하는 것이 가장 현실적이다.

## Key Decisions

- `1차 목표는 JPA/QueryDSL + OpenAPI 도입으로 제한한다.`
  - 이유: 문서와 실제 구현 간 핵심 차이를 줄이면서도 Redis 전환 리스크를 피할 수 있다.

- `Redis는 이번 브레인스토밍의 구현 범위에서 제외한다.`
  - 이유: 기존 계획 문서에서도 Redis는 후속 단계 도입 후보로 정리되어 있었고, 인증/예약/QR의 상태 저장 정책까지 동시에 바꾸면 전환 비용이 너무 커진다.

- `JPA/QueryDSL 전환 범위는 가능한 일반 도메인 전반으로 넓게 잡는다.`
  - 이유: 파일럿 수준으로 끝내면 문서-코드 드리프트를 구조적으로 해소하지 못한다.

- `인증 영역(AuthUser, refresh token 저장소)도 이번 JPA 전환 범위에 포함한다.`
  - 이유: 인증만 JDBC로 남기면 아키텍처 문서와 구현 간 예외가 크게 남고, 저장소 기술 표준화 효과가 약해진다.

- `OpenAPI는 dev 환경에서만 Swagger UI를 노출한다.`
  - 이유: 개발 생산성을 확보하면서 staging/prod 노출 리스크를 피할 수 있다.

- `정산/리포트 같은 복잡한 집계 영역도 이번 전환 대상에 최대한 포함한다.`
  - 이유: 단순 CRUD만 옮기면 “핵심은 여전히 JDBC/수동 SQL” 상태가 남는다. 다만 실제 구현 시 QueryDSL과 native SQL의 혼합 사용은 허용한다.

## Resolved Questions

- `이번 작업의 1차 목표는 무엇인가?`
  - 답: `JPA/QueryDSL + OpenAPI`를 우선 도입하고 Redis는 제외한다.

- `JPA/QueryDSL은 어느 수준까지 전환할 것인가?`
  - 답: 정산/리포트 예외 가능성은 남기되, 가능한 저장소 대부분을 전환 범위에 포함한다.

- `인증 영역도 전환 대상에 포함할 것인가?`
  - 답: 포함한다.

- `OpenAPI는 어디까지 노출할 것인가?`
  - 답: `dev`에서만 Swagger UI를 노출한다.

- `정산/리포트처럼 SQL이 복잡한 영역은 어떻게 볼 것인가?`
  - 답: 최대한 전환 대상으로 포함한다. 단, 구현 시 native SQL 유지 예외는 허용 가능하다.

## Open Questions
