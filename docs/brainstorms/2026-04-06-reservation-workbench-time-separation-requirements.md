# 예약 워크벤치 일정 정보 / 예약 생성 시간 분리

## Problem Frame
현재 예약 관리의 예약 워크벤치에서는 `일정 정보` 칼럼에 `reservedAt`이 표시되고 있어, 사용자가 보고 있는 시간이 "실제 예약 일정"인지 "예약을 생성한 시점"인지 직관적으로 이해하기 어렵습니다. 1차 개선으로 일정 시간과 생성 시간을 시각적으로 분리했지만, 워크벤치가 예약 row와 별도의 미래 스케줄 목록을 조인하는 구조라서 `scheduleId` 매칭이 실패하면 `현재 예약 기준 일정 정보를 확인할 수 없습니다.` 같은 fallback 문구가 노출됩니다. 워크벤치는 현재/예정 예약을 빠르게 운영하는 화면이므로, 예약 행이 존재하는 이상 일정 정보도 최대한 안정적으로 복원되어야 합니다.

## Requirements

**[일정 정보의 우선순위 재정의]**
- R1. 예약 워크벤치의 `일정 정보` 칼럼은 예약 일정의 `startAt ~ endAt`을 가장 우선적인 대표 정보로 표시해야 한다.
- R2. 예약 일정의 시간 아래에는 보조 정보로 트레이너명과 수업명(또는 슬롯명)을 함께 표시해야 한다.
- R3. 기존 `reservedAt` 값은 더 이상 일정 정보의 대표 시간으로 보이면 안 되며, 예약 생성 시각으로 별도 표현되어야 한다.

**[생성 시간 표시 정책]**
- R4. 예약 생성 시간은 `예약 생성: YYYY-MM-DD HH:mm` 형식으로 일정 정보 아래 보조 메타데이터로 표시해야 한다.
- R5. 생성 시간은 숨김, 툴팁, hover 기반 노출이 아니라 기본적으로 항상 보이도록 해야 한다.

**[스케줄 보강 조회]**
- R6. 예약 워크벤치는 예약 row에 포함된 `scheduleId` 목록을 기준으로, 현재 스케줄 목록에 없는 일정도 보강 조회할 수 있어야 한다.
- R7. 스케줄 보강 조회는 기존 스케줄 조회 API를 확장하는 방식으로 해결해야 하며, 워크벤치를 위해 별도 예약 전용 읽기 API를 새로 만들지 않는다.
- R8. 보강 조회는 워크벤치에 표시 중인 예약 행에 필요한 `scheduleId`만 대상으로 해야 하며, 전체 과거 스케줄을 무제한으로 다시 불러오는 방식은 피해야 한다.

**[워크벤치 가독성 및 운영 맥락]**
- R9. 예약 워크벤치의 시간 정보 표현은 운영자가 한 행을 빠르게 훑었을 때 "언제 예약이 잡혀 있는지"와 "언제 생성되었는지"를 혼동 없이 구분할 수 있어야 한다.
- R10. 이 개선은 예약 워크벤치의 기존 액션 흐름과 예약 범위(현재/예정 예약 중심)를 바꾸지 않고 정보 표시 방식만 명확하게 만드는 데 집중해야 한다.
- R11. 보강 조회 이후에도 스케줄 정보를 찾지 못한 경우, 워크벤치는 단순 에러 문구 대신 `삭제되었거나 조회 불가한 일정`처럼 운영자가 의미를 이해할 수 있는 fallback 문구를 표시해야 한다.

## Success Criteria
- 운영자가 예약 워크벤치에서 각 예약의 실제 일정 시간(`startAt ~ endAt`)을 첫눈에 구분할 수 있다.
- 운영자는 예약 생성 시각을 일정 시간과 다른 의미의 보조 정보로 읽을 수 있다.
- 한 셀 안의 시간 정보가 지나치게 많아져 스캔 속도가 떨어지지 않는다.
- 현재 스케줄 목록에 없는 예약이라도, `scheduleId` 기반 보강 조회로 대부분의 워크벤치 행에서 일정 시간이 정상적으로 복원된다.
- 실제로 복원할 수 없는 예약만 명확한 fallback 문구를 노출하고, 사용자 입장에서 이유를 유추할 수 있다.

## Scope Boundaries
- 예약 생성/수정 플로우 자체는 바꾸지 않는다.
- 체크인, 완료, 취소, 노쇼 처리 액션의 동작 방식은 변경하지 않는다.
- 예약 워크벤치 전체 레이아웃 개편이나 별도 상세 모달 추가는 이번 범위에 포함하지 않는다.
- 예약 이력의 전체 타임라인을 새로 만드는 것은 이번 범위에 포함하지 않는다.
- 과거 완료/취소/노쇼 이력을 위한 별도 표기 정책 확장은 이번 범위에 포함하지 않는다.
- 워크벤치 전용 신규 예약 API나 신규 읽기 모델 설계는 이번 범위에 포함하지 않는다.

## Key Decisions
- **일정 우선 표시**: 워크벤치에서 가장 먼저 보여야 하는 시간은 예약 일정 자체의 `startAt ~ endAt`이다.
- **항상 노출되는 생성 시간**: 예약 생성 시간은 `reservedAt`을 활용해 기본 노출한다.
- **현재 예약 중심 해석 유지**: 워크벤치는 현재/예정 예약을 보는 화면으로 유지하고, 과거 이력까지 아우르는 별도 상태별 시간 정책은 이번 범위에서 다루지 않는다.
- **기존 스케줄 조회 API 확장**: fallback 문제는 예약 API enrich나 워크벤치 전용 API 대신, 기존 스케줄 조회 API에 `scheduleIds` 기반 보강 조회 능력을 추가해 해결한다.
- **설명 가능한 fallback 유지**: 복원 실패 행은 숨기지 않고, 운영자가 원인을 이해할 수 있는 구체 fallback 문구를 유지한다.

## Dependencies / Assumptions
- 예약 워크벤치는 예약 row의 `scheduleId`와 예약 스케줄 조회 응답의 `startAt/endAt`, `trainerName`, `slotTitle`를 조합해 일정 정보를 표시할 수 있어야 한다.
- 기존 스케줄 조회 API는 미래 예약 생성 화면에서도 계속 사용되므로, `scheduleIds` 기반 보강 조회 확장은 기존 기본 동작을 깨지 않아야 한다.
