# 비밀번호 분실 복구: 관리자 승인/초기화 운영 플로우

## Problem Frame

사용자가 비밀번호를 잊었을 때, 이번 범위에서는 새로운 self-service 복구 인프라를 만들지 않고 기존 관리자 초기화 절차로만 접근을 복구한다.
이 방식의 목표는 운영 절차를 단순하게 유지하면서도, 사용자가 다시 로그인할 수 있는 명확한 경로를 제공하는 것이다.

핵심 제약은 다음과 같다.
- 백엔드 API를 추가하지 않는다.
- DB 스키마를 변경하지 않는다.
- 요청/승인 대기열을 새로 만들지 않는다.
- 복구 완료 후에도 기존 비밀번호 라이프사이클과 강제 변경 규칙은 그대로 유지한다.

---

## Actors

- A1. 최종 사용자: 비밀번호를 잊어 로그인할 수 없는 사람
- A2. 관리자 운영자: 기존 사용자 계정 관리 화면에서 비밀번호 초기화를 수행하는 사람
- A3. 센터 지원 채널: 사용자의 복구 요청을 관리자에게 전달하는 기존 외부 연락 수단

---

## Key Flows

- F1. 비밀번호 분실 후 복구 요청 전달
  - **Trigger:** 사용자가 로그인에 실패하고 비밀번호를 기억하지 못한다.
  - **Actors:** A1, A3, A2
  - **Steps:** 사용자가 기존 지원 채널로 복구를 요청한다. 지원 채널은 새 앱 기능이 아니라 운영 절차로만 동작한다. 관리자는 기존 사용자 계정 관리 화면에서 대상 계정을 찾아 비밀번호를 초기화한다. 사용자는 임시 비밀번호로 다시 로그인한다.
  - **Outcome:** 사용자가 새 계정이나 새 요청 상태를 만들지 않고도 다시 접근할 수 있다.
  - **Covered by:** R1, R2, R3, R4, R5

- F2. 복구 후 재진입
  - **Trigger:** 사용자가 임시 비밀번호로 다시 로그인한다.
  - **Actors:** A1
  - **Steps:** 사용자는 기존 강제 변경 흐름으로 들어간다. 새 비밀번호를 설정한다. 이후 정상 업무 화면으로 복귀한다.
  - **Outcome:** 복구는 한 번의 관리자 초기화로 끝나고, 최종 비밀번호는 사용자 본인이 다시 확정한다.
  - **Covered by:** R4, R6

---

## Requirements

**복구 방식**
- R1. 비밀번호 분실 복구는 self-service가 아니라 관리자 운영자에 의한 초기화로만 제공한다.
- R2. 사용자는 앱 안에서 복구 요청을 생성하거나 상태를 추적하지 않는다.
- R3. 복구는 기존 사용자 계정 관리에서 이미 제공되는 비밀번호 초기화 동작을 재사용한다.

**사용자 경험**
- R4. 로그인 화면은 비밀번호를 잊은 사용자가 다음 행동을 이해할 수 있도록, 관리자에게 문의하라는 안내를 제공한다.
- R5. 복구 안내는 사용자를 다른 계정이나 별도 요청 화면으로 유도하지 않고, 현재 센터의 운영 절차로 이어지게 해야 한다.

**보안 및 상태**
- R6. 관리자 초기화 후에는 기존과 동일하게 임시 비밀번호 + 강제 변경 흐름을 유지한다.
- R7. 복구는 현재 센터 범위 안에서만 수행되며, 센터 밖의 계정에는 적용되지 않는다.
- R8. 기존 관리자/역할 권한 경계를 바꾸지 않는다.

---

## Acceptance Examples

- AE1. **Covers R1, R3, R4, R6.** 사용자가 로그인에 실패하고 비밀번호를 잊었다고 판단한다. 로그인 화면의 안내를 보고 현재 센터 관리자에게 문의한다. 관리자가 기존 계정 관리 화면에서 비밀번호를 초기화한다. 사용자는 임시 비밀번호로 다시 로그인하고 강제 변경 흐름으로 들어간다.
- AE2. **Covers R2, R5, R7.** 사용자가 앱 안에서 비밀번호 분실 요청을 찾으려 하지만, 별도 요청 생성이나 상태 추적 화면은 없다. 사용자는 현재 센터의 운영 채널로만 안내받고, 다른 센터 계정에는 어떤 복구 처리도 적용되지 않는다.

---

## Success Criteria

- 비밀번호를 잊은 사용자가 새 backend API나 DB 작업 없이 다시 로그인할 수 있는 경로를 이해한다.
- 운영자는 기존 계정 관리 기능만으로 복구를 처리할 수 있다.
- 복구 이후의 비밀번호 정책은 기존 강제 변경 규칙과 충돌하지 않는다.
- 다음 단계 계획 작성 시, 어떤 흐름을 구현해야 하는지 문서가 더 이상 invent하지 않아도 된다.

---

## Scope Boundaries

- 이메일, SMS, MFA, 보안 질문 같은 외부 채널 본인확인 복구는 제외한다.
- 사용자 요청 큐, 승인 대기열, 거절 사유 기록, 복구 이력 조회는 제외한다.
- 새로운 backend API 추가는 제외한다.
- DB 스키마 변경은 제외한다.
- 비밀번호 정책 자체를 바꾸는 일은 제외한다.

---

## Key Decisions

- 복구 모델은 self-service가 아니라 운영자 보조형으로 정한다. 새 인프라를 만들지 않고도 현재 제품 구조에 가장 적은 비용으로 맞는다.
- 기존 관리자 초기화는 복구의 canonical path로 유지한다.
- 복구 완료 후에는 임시 비밀번호와 강제 변경 규칙을 그대로 재사용한다.

---

## Dependencies / Assumptions

- 현재 사용자 계정 관리 화면의 관리자 초기화 기능은 유지된다고 가정한다.
- 사용자는 운영 채널을 통해 관리자에게 접근한다는 외부 절차가 이미 존재한다고 가정한다.
- 로그인 화면에 추가되는 안내 문구는 frontend-only 변경으로 충분하다고 가정한다.
