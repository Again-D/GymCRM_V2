# 회원권 홀딩 개선 및 자동화 요구사항

## Problem Frame
현재 GymCRM 시스템은 홀딩 일수와 횟수 제한 로직이 백엔드에 다수 존재하나, 이를 화면 상에서 직관적으로 파악하거나 유연하게 예외 처리(Override)하기 어려운 상태입니다. 또한, 운영자가 회원권 복귀일(holdEndDate) 도래 시 수동으로 '재개(Resume)' 처리를 해야 하는 번거로움이 있어, 데스크 업무 자동화 차원의 스케줄링이 필요합니다.

## Requirements

**[ 홀딩 UI 및 유연한 정책 제어 ]**
- R1. **잔여 한도 표시**: 프론트엔드의 홀딩 모달에서 해당 회원권 상품이 지원하는 `최대 홀딩 횟수`, `최대 홀딩 일수` 대비 현재 `사용한 횟수`와 `누적 사용 일수`를 노출해야 합니다.
- R2. **소프트 오버라이드 (예외 허용)**: 한도를 초과해 홀딩 날짜를 설정하려 할 경우, 시스템은 강제로 막지 않고(Disabled 속성 미사용) "한도가 초과되었습니다. 계속 진행하시겠습니까?" 와 같은 경고(Soft Warning)만 제공합니다.
- R3. **백엔드 정책 검증 패스 플래그**: 오퍼레이터가 예외를 허용한 경우, API `HoldRequest`에 `overrideLimits` 플래그를 추가로 전송하며, 백엔드는 해당 플래그가 `true`일 때 기존의 `maxHoldDays`, `maxHoldCount` 비즈니스 규칙 검증을 우회 처리해야 합니다.

**[ 시스템 기반 자정 자동 재개 (Auto-Resume) ]**
- R4. **자동 해제 배치**: 지정된 홀딩 종료일(`holdEndDate`) 이 지났거나 당일이 도래했을 때, 데스크 개입 없이 백엔드의 Spring Batch/Scheduler가 수동 `resume()` API와 동일한 방식으로 상태를 `HOLDING -> ACTIVE`로 변경하고 사용 누적 일수를 자동 차감/갱신해야 합니다. 매일 1회 특정한 시간에 스캔하여 실행합니다.
- R5. **재개 알림 발송 (Notification)**: 배치가 회원권의 홀딩 상태를 해제(`ACTIVE` 처리)한 직후, 회원 정보에 등록된 연락처로 "홀딩 자동 해제 및 재개" 안내 메시지(SMS 또는 앱 푸시 등 연동)를 발송하여 회원의 이용 재개를 독려합니다.

## Success Criteria
- 운영자는 고객 대면 시 홀딩 가능 일수 및 횟수를 UI 상에서 오차 없이 확인할 수 있습니다.
- 백엔드 예외 우회 옵션을 통해 VIP 고객 등에 대한 임의 홀딩 기간 연장을 시스템 변경 없이 프론트엔드에서 즉시 조치할 수 있습니다.
- 홀딩 종료일이 지난 회원의 100%가 휴먼 에러나 누락 없이 자동으로 `ACTIVE` 상태로 복원되며 안내 메시지를 수신합니다.

## Scope Boundaries
- **미래 홀딩 예약 상태 미반영**: 당장 '다음달 홀딩 예약'과 같은 별도의 PENDING_HOLD 상태 추가는 제외하여 정산 로직에 주는 충격을 방지합니다. 기존 방식대로 미래일자가 포함되더라도 즉시 예약 당일부터 `HOLDING`으로 넘기는 구조를 당분간 유지합니다.
- **홀딩 히스토리 조회 기능 보류**: 단기 달성 목표에서 제외, 추후 고객 대응 강화를 위한 독립적인 요구사항으로 개발을 미룹니다.

## Key Decisions
- **경고 후 허용(Override) 방식 결정**: 로컬 프랜차이즈나 일반 헬스장의 특성상 매뉴얼 예외(유도리)를 시스템에 강제 차단할 경우 불만율이 올라가므로, Soft Warning으로 해결하기로 결정함.
- **자동-알림 방식 채택**: 매일 자정에 동작하는 알림 메시징 로직 추가를 통해 잠재 회원 이용 복귀율을 상승시키기로 함.
