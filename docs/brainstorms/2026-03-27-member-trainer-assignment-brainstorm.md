# Member-Trainer Assignment

## What We're Building
회원에게 담당 트레이너를 배정하는 운영 흐름을 추가한다. 다만 현재 데이터 모델의 실제 배정 단위는 `member` 자체가 아니라 `member_memberships.assigned_trainer_id`이므로, 1차 범위는 "회원권 신규 등록 시 담당 트레이너를 선택할 수 있게 하는 것"으로 둔다.

이 변경으로 관리자/데스크는 회원권 생성 시 담당 트레이너를 함께 저장할 수 있고, 기존에 이미 구현된 트레이너의 담당 회원 조회 및 예약 권한 흐름이 실제 운영 입력과 연결된다. 트레이너 화면은 우선 조회 중심으로 유지한다.

## Why This Approach
검토한 방향은 두 가지였다.

첫째는 회원권 등록 모달에서 담당 트레이너를 선택하게 하는 방식이다. 이 방식은 현재 백엔드 구매 API가 이미 `assignedTrainerId`를 받고 있고, 예약 권한도 해당 회원권의 배정값을 기준으로 동작하기 때문에 가장 작은 변경으로 즉시 효과가 난다.

둘째는 트레이너 화면이나 별도 배정 화면에서 회원을 배정/변경/해제하는 방식이다. 하지만 현재 저장 구조는 회원권 단위이고, 트레이너 상세 화면도 조회 전용이다. 지금 단계에서 별도 재배정 UI까지 열면 모델과 화면 책임이 동시에 커진다.

추천은 첫째 방식이다. 지금 필요한 것은 "배정값을 입력할 수 있는 최초 경로"이지, 복잡한 재배정 운영 콘솔이 아니다.

## Key Decisions
- 회원권 등록 모달을 1차 배정 입력 지점으로 사용한다: 현재 저장 모델이 회원권 단위이기 때문이다.
- 담당 트레이너는 선택사항으로 둔다: 현재 백엔드도 `assignedTrainerId`가 null 가능하며, 미배정 회원권을 허용하고 있다.
- 트레이너 화면은 1차에서 조회 전용으로 유지한다: 이미 배정 회원 요약을 보여주므로 결과 확인 화면 역할은 충분하다.
- 1차 범위에는 재배정/해제 기능을 넣지 않는다: 저장 경로를 먼저 연결하고 운영 필요가 확인되면 후속으로 확장한다.
- 예약 기능은 별도 수정 없이 기존 권한 흐름을 재사용한다: 배정값이 저장되면 트레이너 대상 조회와 예약 제한이 그대로 작동한다.

## Resolved Questions
- 배정 UI를 어디에 먼저 넣는가: 회원권 신규 등록 모달에 먼저 넣는다.
- 트레이너 상세 화면을 배정 관리 화면으로도 쓸 것인가: 1차에서는 아니다.
- 배정 단위를 회원 기준으로 볼 것인가 회원권 기준으로 볼 것인가: 현재 구현 기준으로 회원권 기준을 따른다.
- 담당 변경이 자주 필요한가: 기본 흐름은 신규 등록 시 배정으로 충분하지만, 고객 요청에 따라 후속 재배정 기능은 필요할 수 있다.
- 한 회원이 여러 PT 회원권을 가질 수 있는가: 현재 구현상으로는 막는 제약이 뚜렷하지 않지만, 운영 정책은 PT 회원권은 가급적 1개만 유지하고 담당 트레이너도 한 명으로 맞추는 방향으로 본다.

## Open Questions
- PT 회원권 중복 보유를 1차 범위에서 단순 경고로 둘지, 생성 차단 정책까지 함께 넣을지 결정이 필요하다.
