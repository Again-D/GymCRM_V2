# AGENTS.md Slimming Brainstorm

## What We're Building
`AGENTS.md`를 약 100줄 내외의 짧은 운영 규칙 문서로 줄이고, 길어지는 설명과 구조 예시는 `docs/` 아래 별도 문서로 분리한다.

핵심 방향은 `AGENTS.md`를 “에이전트가 바로 따라야 하는 절대 규칙 + 핵심 실행 기본값”만 담는 문서로 되돌리는 것이다. 패키지 구조, 긴 배경 설명, 예시 트리, 상세 운영 해설은 별도 문서로 옮겨서 `AGENTS.md`는 짧고 빠르게 읽히는 인덱스형 규칙 문서로 유지한다.

길이 목표는 약 100줄 내외다. 80줄 수준은 토큰 절감에는 유리하지만 필요한 실행 규칙까지 잘릴 위험이 있고, 120줄 이상은 다시 설명이 불어나 문서 분리 효과가 약해진다.

## Why This Approach
현재 [AGENTS.md](/Users/abc/projects/GymCRM_V2/AGENTS.md)는 브랜치 전략, backend/common 구조, 예시 트리, build/test 명령, 문서 sync 규칙까지 모두 담고 있어 실행 문서로는 길어졌다. 사용자 문제의식대로 OpenAI 권장에 더 가깝게 가려면, 첫 화면에 보이는 핵심 규칙만 남기고 나머지는 참조 문서로 보내는 편이 맞다.

완전히 링크만 남기는 초압축 방식보다, `절대 규칙 + 핵심 실행 기본값`은 `AGENTS.md`에 유지하는 것이 낫다. 그래야 에이전트가 별도 문서를 열지 않고도 대부분의 실무 결정을 바로 내릴 수 있다. 상세 구조는 `docs/` 문서로 분리하되, 문서 수는 너무 많지 않게 2개로 유지해 탐색 비용을 줄인다.

## Key Decisions
- `AGENTS.md`는 짧은 운영 규칙 문서로 축약한다.
- `AGENTS.md`에는 다음만 남긴다.
  - 문서 목적
  - 절대 규칙
  - 핵심 branch 전략
  - 핵심 backend/frontend 실행 기본값
  - 상세 문서 링크
- `AGENTS.md`는 “지금 당장 행동에 필요한 규칙”만 담는다.
  - 이유, 긴 배경 설명, 예시 트리, 구조 해설은 상세 문서로 이동한다.
- 패키지 구조, 예시 트리, 긴 설명은 `docs/`로 이동한다.
- 상세 문서는 2개로 나눈다.
  - `브랜치/작업 규칙` 문서
  - `백엔드 구조 규칙` 문서
- `브랜치/작업 규칙` 문서에는 아래를 둔다.
  - 브랜치 전략
  - 작업 단위
  - 검증 기준
  - 문서 sync 규칙
- `백엔드 구조 규칙` 문서에는 아래를 둔다.
  - backend package 구조 기본값
  - `common` 경계
  - feature 패키지 기본 구조
  - 예시 트리와 예외
- 문서를 3개 이상으로 세분화하지 않는다.
  - 지금 목표는 가볍고 찾기 쉬운 구조이지, 문서 체계를 더 복잡하게 만드는 것이 아니다.
- `AGENTS.md`는 인덱스 역할도 하되, 단순 링크 모음으로 만들지는 않는다.
  - 에이전트가 자주 쓰는 규칙은 본문에 남겨둔다.
- 문서 계층은 아래처럼 고정한다.
  - `AGENTS.md`: 최상위 agent-operational source of truth
  - 상세 문서 2개: `AGENTS.md`를 보조하는 참조 문서
  - 상세 문서와 `AGENTS.md`가 충돌하면 `AGENTS.md`를 우선한다
- `AGENTS.md` 길이 목표는 약 100줄 내외로 둔다.
  - 80줄 수준으로 과도하게 줄이지 않는다
  - 다시 120줄 이상으로 불어나지 않게 관리한다

  - 백엔드 구조 규칙 문서
