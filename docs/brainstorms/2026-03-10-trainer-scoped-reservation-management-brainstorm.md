# Trainer Scoped Reservation Management Brainstorm

## What We're Building
`/Users/abc/projects/GymCRM_V2/docs/05_추가_요구사항.md`의 범위를 구현 가능한 단위로 다시 자른다. 요구사항의 핵심은 두 가지다. 첫째, 회원관리 화면에 `트레이너별`, `기간별`, `상품별` 검색 필터를 추가하는 것. 둘째, 예약 관리를 `예약권 보유 회원 리스트` 중심으로 재구성하고, 트레이너 계정에서는 `본인에게 할당된 회원`만 보이도록 만드는 것이다.

이번 브레인스토밍의 목적은 `트레이너 계정`을 같이 도입할 때 어디까지를 이번 단계 범위로 잡는 것이 맞는지 정리하는 것이다.

## Current Repo Reality
현재 저장소 기준으로는 백엔드와 프런트의 준비 상태가 다르다.

- 백엔드는 `ROLE_TRAINER` 자체는 허용한다.
- 설계 문서에는 `assigned_trainer_id` 기반의 담당 회원권 모델이 정의돼 있다.
- 하지만 현재 활성 membership 도메인 구현에는 `assigned_trainer_id`가 아직 반영돼 있지 않다.
- 프런트 auth 타입은 아직 `ROLE_CENTER_ADMIN`, `ROLE_DESK` 중심으로 좁혀져 있다.
- 현재 예약 관리는 `선택된 회원 기준 후속 업무` 구조이지, `예약권 보유 회원 리스트 워크스페이스` 구조는 아니다.

즉, 이번 요구사항은 단순 UI 추가가 아니라 일부 권한/데이터 모델 확장이 같이 필요하다.

## Why This Needs Deliberate Scoping
`트레이너 계정도 같이 도입`이라는 말을 그대로 넓게 해석하면 범위가 급격히 커진다. 트레이너 전용 대시보드, 스케줄 관리, 회원 메모, 정산, 알림 권한까지 이어질 수 있기 때문이다.

하지만 `05_추가_요구사항.md`가 실제로 요구하는 것은 그 전체가 아니다. 지금 필요한 것은 `트레이너가 자기 담당 회원의 예약 업무를 볼 수 있는 최소 권한 경계`다.

따라서 이번 단계는 `트레이너 전용 업무 시스템 전체`가 아니라 `트레이너 스코프 예약 관리`로 제한하는 것이 맞다.

## Approaches Considered
### Approach A: 현재 권한 체계 유지, 관리자/데스크만 먼저 구현
- 회원관리 필터와 예약권 보유 회원 리스트만 먼저 구현한다.
- 트레이너 계정은 이번 단계에서 제외한다.
- 장점: 구현이 가장 빠르고 데이터 모델 변경도 최소화된다.
- 단점: 요구사항의 핵심 차별점인 `트레이너 본인 담당 회원 제한`이 빠진다.
- 판단: 임시 단계로는 가능하지만, 이번 요구사항을 제대로 닫는 접근은 아니다.

### Approach B: 트레이너 계정 전체 도입
- 트레이너 로그인, 트레이너 전용 메뉴, 스케줄 편집, 담당 회원 조회, 예약 조정, PT 일지, 정산, 알림까지 한 번에 설계한다.
- 장점: 역할 체계를 한 번에 정리할 수 있다.
- 단점: 현재 요구사항 대비 과도하고, 데이터 모델/권한/UI 범위가 너무 넓다.
- 판단: 지금 단계에서 과하다. 실패 가능성이 높다.

### Approach C (Selected): 트레이너 스코프 예약 관리 최소 도입
- `ROLE_TRAINER`를 프런트 로그인/세션에 활성화한다.
- 회원권에 `assigned_trainer_id`를 추가해 담당 회원 기준을 만든다.
- 예약 관리를 `예약권 보유 회원 리스트` 중심으로 재구성한다.
- 관리자 계정은 전체 예약권 보유 회원을 본다.
- 트레이너 계정은 `본인에게 할당된 회원 + 예약권 보유` 조건만 본다.
- 회원 상세 모달에서 회원정보, 상품정보, 예약상태, 예약일정을 본다.
- 예약 조정은 정책상 허용되는 상태/시간 범위에서만 수행한다.
- 장점: 요구사항의 핵심을 충족하면서 범위를 제어할 수 있다.
- 단점: membership 데이터 모델과 RBAC 테스트까지 같이 건드려야 한다.

## Selected Direction
이번 단계는 `트레이너 전용 포털`을 만드는 것이 아니라 `트레이너 스코프 예약 업무`를 도입하는 것으로 제한한다.

구현 범위는 아래까지로 자른다.

1. `ROLE_TRAINER` 세션/프런트 활성화
2. membership에 `assigned_trainer_id` 추가
3. 회원관리 상단 필터 확장
4. 예약 관리를 `예약권 보유 회원 리스트 + 검색 필터 + 상세 모달` 구조로 재구성
5. 역할별 리스트 범위 차등 적용
6. 예약 조정 정책을 프런트와 백엔드 모두에서 강제

## What "Trainer Included" Means In Practice
이번 단계에서 `트레이너 계정도 같이 도입`은 아래 의미로 해석한다.

- 트레이너 로그인 가능
- 트레이너는 본인 담당 회원 중 예약권 보유 회원만 예약 관리에서 조회 가능
- 해당 회원의 예약 현황과 일정 확인 가능
- 허용 범위 내 예약 취소/노쇼/일정 조정 가능

이번 단계에 포함하지 않는 것은 아래다.

- 트레이너 전용 대시보드
- 트레이너 스케줄 생성/편집 UI 전체
- PT 수업 일지/상담 기록
- 트레이너 정산 화면
- 회원-트레이너 자동 배정 로직
- 알림 정책 확장

## Member Management Filter Direction
회원관리 상단 필터는 `전체 회원 검색` 위에 업무 필터를 얹는 구조가 적절하다.

필터 후보:
- 담당 트레이너
- 기간
- 상품

주의할 점:
- `기간`은 가입일/회원권 시작일/예약일 중 무엇인지 의미를 고정해야 한다.
- 현재 요구사항 문맥상 `회원관리` 필터이므로 우선 `회원권 시작일 또는 구매일` 기준으로 해석하는 쪽이 자연스럽다.
- 상품/트레이너 필터는 결국 membership join이 필요하므로 member summary query 확장이 필요하다.

## Reservation Workspace Direction
현재 예약 관리는 `선택된 회원`을 전제로 한 후속 업무 탭이다. 이번 요구사항은 이를 `예약권 보유 회원 리스트 워크스페이스`로 한 단계 올리는 쪽이 맞다.

추천 구조:
- 좌측 또는 상단: 예약권 보유 회원 리스트 + 검색 필터
- 우측 또는 본문: 선택 회원 상세/예약 일정/조정 액션
- 회원명 클릭 시 상세 모달 제공
- 이미 있던 direct-entry member picker 패턴은 예약권 보유 회원 리스트 구조에 흡수

이 구조가 좋은 이유는 트레이너 계정에서도 같은 화면을 쓰되, 데이터 범위만 다르게 적용할 수 있기 때문이다.

## Reservation Adjustment Policy
정책은 UI 안내만으로 끝내면 안 된다. 백엔드에서도 막아야 한다.

필수 규칙:
- 과거 일정은 수정 불가
- `NO_SHOW`는 수업 종료 후에만 가능
- 이미 취소/완료/노쇼된 예약은 다시 조정 불가
- 현재 진행 중이거나 미래 일정만 조정 가능
- 트레이너는 본인 스코프 밖 예약을 조회/조정할 수 없어야 한다.

## Data Model Implications
핵심 데이터 모델 변화는 membership에 담당 트레이너를 연결하는 것이다.

최소 변경:
- `member_memberships.assigned_trainer_id` 추가
- 관련 인덱스 추가
- membership 생성/수정 API에 trainer assignment 반영
- reservation/member summary query에서 trainer scope 필터 가능하게 확장

중요한 점:
- `트레이너 담당 회원`을 member 자체에 붙일지 membership에 붙일지는 선택해야 한다.
- 현재 요구사항은 `예약권 보유 회원`과 결합돼 있고, 설계 문서도 membership 기준이므로 이번 단계는 membership에 붙이는 쪽이 더 맞다.

## Risks To Avoid
- 트레이너 스코프 정의 없이 프런트 필터만 넣어 데이터 노출 경계가 흐려지는 것
- member와 membership 양쪽에 담당 트레이너 개념이 중복돼 ownership이 갈라지는 것
- 예약 조정 가능 여부를 UI만 막고 백엔드가 허용하는 것
- 이번 요구사항을 계기로 트레이너 전용 모듈 전체를 같이 벌여 범위가 무너지는 것
- 예약 관리가 다시 `회원 선택 후 업무`와 `회원 리스트 워크스페이스` 사이에서 애매한 상태가 되는 것

## Recommended Delivery Slice
가장 현실적인 순서는 아래다.

### Slice 1
- `ROLE_TRAINER` 프런트 세션 활성화
- membership에 `assigned_trainer_id` 도입
- 회원관리 필터에 트레이너/상품/기간 추가

### Slice 2
- 예약 관리를 `예약권 보유 회원 리스트` 중심 화면으로 재구성
- 관리자 전체 조회 / 트레이너 스코프 조회 분리
- 회원 상세 모달 추가

### Slice 3
- 예약 조정 정책 고도화
- RBAC/교차센터/과거일정/노쇼 타이밍 테스트 추가

## Success Criteria
- 관리자 계정은 전체 예약권 보유 회원 리스트를 본다.
- 트레이너 계정은 본인 담당 회원 중 예약권 보유 회원만 본다.
- 회원명 클릭으로 상세 모달에서 회원/상품/예약 상태/예약 일정을 볼 수 있다.
- 허용되지 않는 예약 조정은 프런트와 백엔드 모두에서 막힌다.
- 요구사항 범위 밖인 트레이너 전용 전체 포털 구현으로 번지지 않는다.

## Open Questions
- 회원관리의 `기간별` 필터는 `가입일`, `회원권 구매일`, `회원권 시작일` 중 어느 기준인가?
- 담당 트레이너 지정은 회원권 생성 시 필수인가, 선택인가?
- 트레이너가 예약 `일정 조정`까지 가능한지, 아니면 `취소/노쇼`만 가능한지 정책을 더 좁혀야 하는가?

## Resolved Questions
- `트레이너 계정`을 같이 도입하더라도 범위는 `트레이너 스코프 예약 업무`로 제한한다.
- `담당 회원` 경계는 member가 아니라 membership 기준으로 잡는 것이 더 자연스럽다.
- 이번 단계에서는 트레이너 전용 대시보드/정산/PT일지까지 확장하지 않는다.
- 예약 관리는 `예약권 보유 회원 리스트 워크스페이스` 방향으로 재구성하는 것이 맞다.
