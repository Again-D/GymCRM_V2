# Role Model Alignment With Database Design

## What We're Building

현재 구현은 `users.role_code` 단일 컬럼을 권한의 소스 오브 트루스로 사용하지만, [03_데이터베이스_설계서.md](/Users/abc/projects/GymCRM_V2/docs/03_데이터베이스_설계서.md)는 `roles`와 `user_roles`를 통한 역할 매핑 구조를 정의하고 있다. 이번 작업의 목적은 설계서와 실제 구현을 정렬해, 권한 모델의 저장 구조를 `roles` + `user_roles`로 전환하는 것이다.

이번 범위는 전면적인 제품 기능 확장이 아니라 권한 저장 구조와 인증 계약 정렬에 집중한다. 우선순위는 백엔드/DB/JWT 계약을 새 구조로 옮기는 것이고, 프론트는 이 계약 변경을 수용하기 위한 최소 수정만 수행한다. 화면별 세밀한 권한 UX 정리는 후속 백로그로 남긴다.

## Why This Approach

검토한 선택지는 세 가지였다.

### Approach A: 설계서를 현재 구현에 맞게 수정

구현은 그대로 두고 문서만 `users.role_code` 중심으로 바꾸는 방식이다.

**Pros:**
- 가장 빠르다.
- 운영 리스크가 작다.

**Cons:**
- 설계 문서가 의도한 권한 모델 확장성을 포기한다.
- 이미 누적된 구현-문서 불일치를 해소하지 못한다.

**Best when:** 시스템 구조 변경 의지가 없고 문서 정합성만 빠르게 맞추고 싶을 때

### Approach B: 설계서대로 `roles` + `user_roles`로 완전 전환, 실제 다중 역할도 즉시 허용

저장 구조뿐 아니라 권한 해석도 다중 역할 기준으로 즉시 바꾸는 방식이다.

**Pros:**
- 설계서와 가장 완전하게 일치한다.
- 장기 확장성 면에서는 가장 깔끔하다.

**Cons:**
- 현재 코드가 전부 단일 역할 전제라 영향 범위가 너무 크다.
- JWT, Spring Security, 프론트 route guard, 테스트 fixture를 한 번에 재설계해야 한다.

**Best when:** 제품적으로 다중 역할이 당장 필요한 경우

### Approach C: `roles` + `user_roles`로 단일 전환하되, 운영 규칙은 사용자당 1역할 유지

DB 구조는 설계서대로 바꾸지만, 애플리케이션 규칙은 당분간 “사용자 1명 = 역할 1개”로 유지하는 방식이다.

**Pros:**
- 설계서와 저장 구조를 정렬할 수 있다.
- 현재 서비스/프론트 권한 분기를 다중 역할까지 한 번에 재설계하지 않아도 된다.
- 후속 다중 역할 확장을 위한 기반을 남긴다.

**Cons:**
- M:N 스키마를 쓰면서도 운영 규칙은 1역할이라 약간의 개념적 긴장이 남는다.
- 제약 조건과 검증 로직을 명확히 정의해야 한다.

**Best when:** 설계서 정렬이 중요하지만 현재 제품의 권한 해석 복잡도는 통제하고 싶을 때

추천 및 사용자 선택은 Approach C였다.

## Key Decisions

- 저장 구조는 설계서대로 `roles` + `user_roles`로 전환한다.
  - 이유: 현재 문서와 구현의 불일치를 해소하고, 장기적으로 역할 모델 확장 기반을 확보한다.

- 운영 규칙은 당분간 사용자당 1역할만 허용한다.
  - 이유: 현재 코드 전반이 단일 역할 전제를 두고 있어, 다중 역할까지 즉시 열면 전환 범위가 과도하게 커진다.

- 전환 방식은 점진 병행이 아니라 단일 전환으로 본다.
  - 이유: `users.role_code`를 오래 병행하면 이중 소스 오브 트루스가 생기고, 이후 정리가 더 어려워진다.

- 이번 작업의 1차 범위는 백엔드/DB/JWT 계약 정렬이다.
  - 이유: 권한 모델 전환의 핵심 리스크는 저장/인증/인가 계층에 있으며, 프론트는 새 계약을 수용하는 최소 수정만 우선 반영한다.

- 남은 프론트 권한 정리는 후속 백로그로 분리한다.
  - 이유: 현재 프론트는 `authUser.role` 단일 문자열 계약으로 충분히 동작하고 있으므로, 1차 전환에서는 계약 호환성만 확보하는 편이 낫다.

## Resolved Questions

- `user_roles`가 M:N 구조여도 실제 운영에서 다중 역할을 즉시 허용할지?
  - 결정: 아니다. 당분간 사용자당 1역할 운영 규칙을 유지한다.

- `users.role_code`와 `user_roles`를 병행하는 점진 전환으로 갈지?
  - 결정: 아니다. 단일 전환을 목표로 한다.

- 이번 전환 범위를 어디까지 볼지?
  - 결정: 백엔드/DB/JWT 계약 우선, 프론트는 최소 수정만 포함한다.

## Backlog After Cutover

- 프론트 권한 모델을 `roles`/배열 기반 계약까지 확장할지 재평가
- 관리자 화면의 role 편집 UX를 `user_roles` 구조에 맞게 재설계
- 테스트 fixture 생성 유틸을 `roles`/`user_roles` 기반 공통 헬퍼로 통합
- 장기적으로 실제 다중 역할 허용이 필요한지 제품 정책 재확인
